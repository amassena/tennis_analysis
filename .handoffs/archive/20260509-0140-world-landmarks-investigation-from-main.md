# Goal

Test the hypothesis that surfaced from `feature/detection/pose-experiments`:

> If biomech outputs feel wrong to the user, the load-bearing weakness is
> almost certainly **not** MediaPipe's 2D pose detection — it's MediaPipe's
> synthesized `world_landmarks` (the model's internal 2D→3D lifting).

Verify this with cheap diagnostic work first; if confirmed, evaluate whether a
purpose-built 2D→3D lifting head (MotionBERT, HMR2) on top of MediaPipe's 2D
output produces more stable biomech outputs than MediaPipe's native
world_landmarks.

This is a **diagnosis-and-targeted-fix** brief, not another model bake-off.

# Why this is the right next stream

Pose-experiments showed that MediaPipe, Sapiens-2 0.4B, and RTMPose-Wholebody
all place 2D keypoints in the same locations to within ~5 pixels at 1080p
(normalized px Δ 0.005). The 2D extractor isn't the bottleneck.

But the user's stated concern was real ("I'm not convinced our pose detection
model is good enough"). Likely sources of the bad-quality signal they're
seeing, in order of probability:

1. **MediaPipe's `world_landmarks` synthesis** — the model's internal
   2D→3D lifting. Synthesized depth coordinates can be wonky during
   motion, occlusion, or unusual poses (which tennis has plenty of:
   serves, contorted recovery shots, etc.).
2. **Biomech equations being brittle** — `biomechanical_analysis.py`
   computes angles and timings in 3D body-relative coords. Under realistic
   3D noise, some metrics (kinetic chain timing, trunk rotation) may
   produce implausible values that look like "pose is bad" but are
   actually "math is amplifying small 3D errors."
3. **Coverage gaps under motion blur** — MediaPipe drops detection on
   left elbow/wrist during fast swings (Sapiens-2 was visibly more
   confident on those frames in the earlier eval). This is a 2D-detection
   issue but only for ~10–20% of swing frames.

This brief targets #1 + #2 directly (+ skirts #3).

# Background — what we have

- **Production MediaPipe poses** at `poses_full_videos/{vid}.json`. Each detected frame has both `landmarks` (2D normalized image coords, with visibility) and `world_landmarks` (3D meters in body-relative frame, with visibility).
- **GT shots** at `detections/{vid}_fused.json`. Per-shot type (forehand/backhand/serve), center frame, audio cues.
- **Biomech analysis** at `scripts/biomechanical_analysis.py`. Reads world_landmarks first, falls back to 2D landmarks. Computes per-shot angles/timings.
- **Coaching outputs** at R2 `highlights/{vid}/coaching.json` — the user-visible product. Generated by `claude_coach.py` from biomech metrics.
- **Locked corpus** of 6 videos with 666 GT shots from sapiens-eval (frozen criteria document).

# Phased experiment plan

## Phase 1 — Audit (~half day, no GPU)

**Goal:** find concrete examples where biomech reports an implausible value, then trace whether MP's 2D pose looks right at that frame but the 3D world_landmarks don't.

1. Pick 3-4 GT videos with a mix of shot types (e.g. IMG_0996, IMG_6705, IMG_6874).
2. Run `biomechanical_analysis.py` on them; flag shots where any of:
   - `knee_angle_at_contact` outside [120°, 180°] (humans don't hyperextend; full-bend is rare at contact)
   - `trunk_rotation_at_contact` outside [-90°, +90°]
   - `arm_extension_at_contact` outside [60°, 180°]
   - `kinetic_chain_timing_ms` with reversed order (wrist before shoulder, etc.)
   - Any value reported as exactly 0 or 180 (degenerate signal)
3. For each flagged shot, save: frame screenshot, MP 2D keypoints overlaid, MP world_landmarks visualization (project 3D body-frame onto a side-view diagram).
4. Manual inspection (~30 shots, ~30 min): is the 2D pose visually correct? Is the 3D body-frame plausible?
5. **Verdict from Phase 1**:
   - "2D looks right, 3D looks wrong on most flagged shots" → strong signal that world_landmarks is the bottleneck, proceed to Phase 2.
   - "Both look wrong" → 2D detection IS sometimes the issue (the motion-blur coverage gaps); narrower follow-up needed.
   - "Both look right; biomech math is amplifying small noise" → the bottleneck is the equations, not pose at all. Different brief: refactor biomech.
   - "Most flagged shots have biomech actually correct, just outside our heuristic ranges" → the user's diagnosis was off-target. Surface to user and pause.

## Phase 2 — Test alternative 3D lifting (gated on Phase 1)

**Only run if Phase 1 confirms world_landmarks as the bottleneck.**

Two candidates:

| # | Approach | What it does | Effort |
|---|---|---|---|
| 2A | **MotionBERT** (Zhu et al, ICCV 2023) | Transformer-based lifter; takes a sequence of 2D keypoints, outputs 3D pose per frame. ~50ms/frame on RTX 4080. | ~1 day |
| 2B | **HMR2** (Goel et al, CVPR 2023) | Vision-foundation lifter; takes single image + 2D keypoints, outputs SMPL body model with 3D joints. Heavier. ~150ms/frame. | ~1.5 days |

For each:
1. Install on tmassena (separate venv if needed; both are PyTorch-based)
2. Write extractor that reads MP 2D output and emits new `world_landmarks` in same schema.
3. Run on the audit-flagged shots from Phase 1.
4. Compare biomech outputs: does the alt-lifter produce more plausible / stable values than MP's native world_landmarks?

## Phase 3 — Decide (gated on Phase 2)

- If alt-lifter clearly better: write production-swap brief (replace MP's world_landmarks with alt-lifter output downstream of pose extraction).
- If alt-lifter not clearly better: world_landmarks isn't trivially fixable; biomech equations may need to be rewritten 2D-only or made noise-robust. Different brief.
- If marginal: dual-mode (MP for fast pipeline, alt-lifter for "high-quality" mode).

# Decision criteria — what counts as "world_landmarks is the bottleneck"

Locked at experiment start (do not edit post-hoc):

- **CONFIRMED weakness if** ≥ 60% of audit-flagged shots show plausible 2D pose but implausible 3D body-frame. (Threshold deliberately above 50% to avoid coin-flip results.)
- **NOT THE BOTTLENECK if** ≤ 30% of flagged shots fit that pattern (most flagged shots have either both 2D + 3D wrong, or both right with biomech math amplifying noise).
- **AMBIGUOUS** in between → look harder; may need GT 3D pose annotations or specific physics-based sanity checks.

For Phase 2 (if reached):

- **ADOPT alt-lifter if** stable biomech metric std-dev across same-shot-type drops by ≥ 20% across all six locked videos vs MP's world_landmarks.
- **REJECT alt-lifter if** std-dev increases or stays within 10%.
- **REVISIT** if mixed (lifts some metrics, breaks others).

# Risks

1. **Phase 1 may show "biomech math is the issue, not 3D"** — that's actually a fine outcome (focuses the next brief precisely) but invalidates Phase 2 spend.
2. **MotionBERT and HMR2 are SMPL-based** in some configurations — output joint conventions differ from MediaPipe's 33-keypoint COCO-derivative. Need a careful keypoint-mapping layer like we built for Sapiens-2 and RTMPose. Reuse the SOCIOPTICON_TO_MEDIAPIPE pattern.
3. **No GT 3D pose annotations** — accuracy claims would be relative-to-MP, not absolute. Phase 1's audit relies on visual plausibility, which is qualitative. If the user wants stronger evidence, ~1 day of frame-level GT labeling (10–20 shots × 5 frames each = ~100 frames) would help.
4. **The locked-criteria conf-delta methodology bug from pose-experiments still applies** to any cross-extractor accuracy comparison. This brief avoids it by comparing biomech-output stability instead, but be careful about backsliding.
